Circuit Breaker Pattern

Prevent a network or service failure from cascading to other services. The circuit breaker pattern stops cascading failures and allows failing services time to recover.

Overview

In a distributed environment, calls to remote resources and services can fail due to transient faults, such as slow network connections, timeouts, or the resources being overcommitted or temporarily unavailable. These faults typically self-correct after a short period, and a robust cloud application should be prepared to handle them by using a strategy such as the **Retry pattern**.

However, there can also be situations where faults are due to unexpected events, and that might take much longer to fix. The **Circuit Breaker** pattern is different from the Retry pattern. The Retry pattern enables an application to retry an operation in the expectation that it'll succeed. The Circuit Breaker pattern prevents an application from performing an operation that is likely to fail.

Metric Complexity Note
Time Complexity O(1) State check and transition are constant time.
Space Complexity O(N) N = number of monitored service endpoints.

Implementation (Go)

GO / Standard Library 

type CircuitBreaker struct {
    threshold  int
    timeout    time.Duration
    failureCnt int
    state      string // "CLOSED", "OPEN", "HALF-OPEN"
    lastFailed time.Time
}

func (cb *CircuitBreaker) Call(fn func() error) error {
    if cb.state == "OPEN" {
        if time.Since(cb.lastFailed) > cb.timeout {
            cb.state = "HALF-OPEN"
        } else {
            return errors.New("circuit open")
        }
    }
    
    err := fn()
    if err != nil {
        cb.failureCnt++
        if cb.failureCnt >= cb.threshold {
            cb.state = "OPEN"
            cb.lastFailed = time.Now()
        }
        return err
    }

    "CLOSED"
    cb.failureCnt = 0
    return nil
}
